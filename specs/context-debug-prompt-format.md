# Spec: Context Debug Prompt Format

## Purpose
Define the standardized format for context debug prompts when transitioning between chat sessions.

## Scope
- In-scope: Technical context formatting, actionable information structure
- Out-of-scope: Personal preferences, memory references, style guidelines

## Core Principle
**Technical context only** - Focus on actionable information for new chat sessions without personal preferences or style guidelines.

## Required Sections

### 1. Problem Statement
- Core technical issue being addressed
- Primary objective of the work
- Context of the problem domain

### 2. Key Architectural Concepts
- Important technical principles
- System architecture decisions
- Core business logic concepts

### 3. Current State & Issues
- What's currently working
- What's broken or needs fixing
- Known limitations or constraints

### 4. Files Involved
- Key files and their roles in the system
- Component hierarchy and relationships
- Data flow between components

### 5. Successfully Fixed Issues (KEEP THESE)
- List of working solutions that should be preserved
- Specific fixes that resolved problems
- Code changes that improved functionality

### 6. Key Technical Details
- Important technical information
- Configuration details
- Integration points and APIs

### 7. Next Steps
- Actionable items for the new session
- Specific tasks to complete
- Testing and validation requirements

## Excluded Sections

### ❌ Do Not Include
- Important Notes
- Memory References
- User preferences
- Style guidelines
- Personal communication preferences
- AI behavior instructions

## Format Guidelines

### Structure
- Use clear section headers with ##
- Bullet points for lists
- Code blocks for technical details
- Focus on technical context only

### Content Focus
- Technical implementation details
- System architecture
- Code structure and relationships
- Problem-solving approaches
- Actionable next steps

### Language
- Direct and technical
- No personal preferences
- No style guidance
- Focus on what needs to be done

## Example Template

```markdown
## Problem Statement
[Core technical issue description]

## Key Architectural Concepts
[Important technical principles]

## Current State & Issues
- Working: [List of working features]
- Broken: [List of issues to fix]

## Files Involved
- [File path]: [Role and purpose]
- [File path]: [Role and purpose]

## Successfully Fixed Issues (KEEP THESE)
1. [Fix description and implementation]
2. [Fix description and implementation]

## Key Technical Details
[Important technical information]

## Next Steps
1. [Specific actionable task]
2. [Specific actionable task]
```

## Acceptance Criteria

### ✅ Include
- Technical problem description
- System architecture details
- Working solutions to preserve
- Specific files and components
- Actionable next steps

### ❌ Exclude
- Personal preferences
- Communication style guidelines
- Memory references
- AI behavior instructions
- Non-technical context
